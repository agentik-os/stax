# U-034: backend continuity — the programmatic data scan + the data check gate

TARGET: {{TARGET}}
REFERENCE: frameword/packages/stax-migrate (datascan.mjs, tests/fixtures/*)

## DETECT
- The data matrix is inventoried BY THE AGENT (prose crawling): slow, partial,
  and unprovable — a table or RPC the agent never noticed simply vanishes from
  the migration.
- Nothing verifies, after mapping, that the CODE and the MATRIX still agree:
  a table added mid-migration ships unbound.
- Convex/Supabase specifics (RLS lockouts, internal functions, storage buckets
  vs tables, dynamic table names) are handled by vibes.

## APPLY
1. THE 80/20 LAW: extraction and verification are PROGRAMMATIC; the AI only
   maps. `stax-migrate data scan --write` reads the SOURCE (never the network)
   and emits D-rows with file:line evidence for: Convex (defineTable, exported
   query/mutation/action/httpAction, internal* listed-not-gated, app call
   sites, ctx.db table ops, UNUSED flags), Supabase (migration DDL incl.
   quoted/schema-prefixed names, RLS + policy counts with a lockout warning at
   zero policies, .from() op sites with storage.from excluded, .rpc, .channel,
   edge functions, dynamic .from flagged), Prisma models + client ops, REST
   route methods, tRPC procedures (first .query/.mutation in the chain wins).
2. MERGE IS SAFE: re-scanning preserves ids and the AI's panel_binding /
   write_path / status, refreshes source/ops/evidence, keeps manual rows and
   flags them stale.
3. THE GATE: `stax-migrate data check` exit-1s while any non-internal row has
   no panel_binding and no REASONED skip (a bare "skipped" is not a reason),
   any writable row (c/u/d/w) lacks its write_path, or the code has a surface
   the matrix does not (drift). DATA 100% is printed only when the whole
   backend is bound to the new front end.
4. PIPELINE ORDER: phase 2 runs the scan FIRST, the agent adds only the
   unprovable; phase 4 maps; `data check` closes the loop before phase 7 ships.

## VERIFY
- On a Convex fixture: 3 tables + query/mutation extracted, internal excluded
  from the gate, the never-called mutation carries "UNUSED by the app".
- On a Supabase fixture: quoted/schema-prefixed DDL parsed, RLS-zero-policy
  warns, storage bucket never counted as a table, dynamic .from warned.
- CLI loop: scan --write → check exit 1 naming every unbound row → fill
  mappings → check prints DATA 100% exit 0 → add a new .from("x") to the code
  → check exit 1 naming the drift.
