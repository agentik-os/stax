# U-009 — Flat everything: in-flow content loses its boxes

TARGET: {{TARGET}}
REFERENCE: frameword/apps/crm-specimen/src/styles.css (.card, .stats/.stat, .drills/.drill,
.anat-row, .leaf-note, .codeblock, .pf-list rows)

## WHY
After U-002 removed FORM cards, doc/content surfaces still wore borders. The rule
generalizes: borders belong to FLOATING surfaces (menus, popovers, pickers), INPUTS and
MEDIA frames — every in-flow element is flat, structured by hairlines and rhythm.

## DETECT
- `grep -n "border: 1px solid" <styles>` on: doc cards, stat tiles, drill rows,
  definition rows, note blocks, list rows (people/services/nav/hub cards).

## APPLY
1. Doc card → flat block: hairline TOP, padding 14/0/15, mono label (mb 8), no bg/radius.
2. Stat band → no tiles: vertical hairline BETWEEN stats, padding-right/left 18.
3. Drill list → hairline top on the list + hairline between rows, hover = secondary fill,
   radius 0; see-also keeps a dashed bottom hairline.
4. Definition rows (anat) → hairline top per row, padding 10/0, no box.
5. Note blocks → hairline top, no box. Code blocks → secondary bg, NO border.
6. List rows (members, services, alerts, nav/hub cards) → hairline list pattern:
   container hairline top, row hairline bottom, hover fill; interactive nav tiles keep
   radius for the hover fill only.
7. KEEP borders on: floating menus/popovers/pickers, inputs, media tiles, semantic
   banners/alert boxes.

## VERIFY
- Screenshot a doc panel, a stats panel, a drill list and a member list, BOTH themes —
  zero in-flow boxes.
- Popovers/menus/inputs still bordered (spot-check three).
